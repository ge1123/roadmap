# Repository Guidelines

## 專案結構與模組組織
本倉庫聚焦於學習路線規劃，全數內容皆為 Markdown。根目錄的 `README.md` 提供主題清單與索引，各領域的詳細計畫集中在 `doc/` 目錄內（例如 `doc/csharp_plan.md`、`doc/design_patterns_plan.md`）。新增資料時請放入 `doc/`，並維持以小寫加底線命名的檔案慣例，確保閱讀與檔案排序一致。

## 建置、測試與開發指令
專案不需編譯，但提交前請執行下列檢查：
- `npx markdownlint-cli2 README.md "doc/**/*.md"`：檢查標題層級、清單格式與連結樣式。
- `npx textlint doc/**/*.md`：若有安裝，協助辨識常見語句與標點問題。
建議透過 `npx` 執行以共用相同設定，必要時可將指令寫入個人腳本以便快速呼叫。

## 程式風格與命名慣例
沿用標準 Markdown 標題階層（`#` → `##` → `###`），保持段落精簡易掃描。流程或重點列表使用無序清單，命令與關鍵詞以反引號標示。新撰寫的學習計畫請以與檔名相符的 H1 作為開頭（例如 `# Clean Architecture Plan`）。文字敘述優先使用精簡的台灣繁體中文，並盡量避免半形、全形標點混用。

## 測試與驗證準則
每次變更後請手動點擊內文連結，確認跨檔案引用（如 `csharp_plan.md` → `testing_plan.md`）仍然有效。若加入程式碼片段，務必事先在對應的環境中驗證可執行，並於內容內註明使用的語言版本或框架，以利讀者重現。

## Commit 與 Pull Request 指南
遵循既有紀錄的祈使句、聚焦主題命名方式（例如：`outline distributed cache roadmap`）。建立 PR 時請摘要本次擴充的知識主題、列出修改過的 Markdown 檔案，並備註新增的工具、引用來源或圖表產出方式，讓審查者能快速比對與追蹤後續更新。
